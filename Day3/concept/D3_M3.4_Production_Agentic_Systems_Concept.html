Concept A
Two Kinds of Memory: Conversations vs. People
You already built one of them this morning and may not have noticed.
✅ SHORT-TERM — you have this already
The checkpointer from M3.2. It remembers everything about one conversation — keyed by thread_id. Pause, resume, crash, recover: all one thread.
Question it answers: "where was this conversation?"
🆕 LONG-TERM — you add this in Task 1
The store — a separate little database of facts keyed by who, not by which conversation. Priya calls today, Priya calls next week: two thread_ids, one Priya.
Question it answers: "what do we know about this person?"

📓 The whole API is three ideas

A namespace (a folder name, like ("customers",)), a key (who — "priya"), and a value (a plain dict of facts). Put. Get. That's it.

from langgraph.store.sqlite import SqliteStore memory_store.put(("customers",), "priya", {"last_intent": "upi_transfer", "interactions": 1}) memory_store.get(("customers",), "priya") # → her facts, from ANY conversation memory_store.get(("customers",), "nobody") # → None — a first-time customer

Concept B
Guardrails: The Agent's Seatbelts
One guardrail is a hope. Layers of guardrails are a design. You'll add three — then attack your own agent to prove one works.
🚪
1 · Input guardrail
Plain regex screening BEFORE any LLM sees the message. Deterministic, auditable, and free — a blocked attack costs zero tokens.
🧱
2 · Hard limits in code
Business rules that live in Python, not prompts. No clever wording can negotiate with an if amount > ceiling.
🎭
3 · Output guardrail
Masks PII — account numbers, emails — in anything customer-facing, no matter which path produced it.

💡 Plot twist: you already built guardrail #4 this morning

M3.2's route_by_risk — the Python function that forces a human approval above ₹1,00,000 — was a guardrail all along: a hard limit in code plus a human gate. Today you add a second threshold to it: above ₹10,00,000, not even a human click is enough — the workflow refuses outright and asks for a branch visit. Some doors an autonomous system should simply not have.

# Before any LLM call — deterministic, explainable to an auditor, free: INJECTION_RE = re.compile("ignore\\s+...instructions|system\\s+override|pretend\\s+you\\s+are|...", re.I) def input_guardrail_node(state): if INJECTION_RE.search(state["customer_message"]): return {"decision": "blocked_by_guardrail", ...} # never reaches the LLM return {"log": ["input_guardrail: message clean"]}

Honest note: regex catches known phrasings, cheaply. Real deployments layer it with an LLM-based screen for novel attacks — same idea as your judge in Concept D. Layers, not silver bullets.

🤔 Predict: why enforce the ₹10,00,000 ceiling in route_by_risk_hardened (Python) instead of adding "never approve amounts above 10 lakh" to the prompt?
B. An LLM instruction is something the model usually follows — and "usually" is exactly what the Chevrolet attackers exploited. An if statement follows nothing but the numbers. Rule of thumb for regulated industries: limits live in code; the LLM only ever proposes, never disposes. (It does save tokens too — but that's a bonus, not the reason.)

Concept C
Observability: See Every Step Your Agent Takes
When run #4,812 of 50,000 does something strange at 2 a.m., what will you look at?

🔍 Two words, one picture

A trace is the complete recording of one run, start to finish. A span is one step inside it — one node, one LLM call, one retrieval. Spans nest, so a trace reads as a tree. Every span carries its own timing, token counts, and the exact prompt in / answer out.

Try it — a real-shaped trace of your triage workflow
Click spans to open them. Mini-game: find the most expensive span in the tree.

🐦 Phoenix — what you run in the lab

Open source (Apache 2.0), from Arize. One pip install, runs a real trace UI at localhost:6006, entirely on your machine — no account, no signup, nothing leaves your laptop. Two lines to attach:

import phoenix as px px.launch_app() from phoenix.otel import register register(project_name="meridian-triage", auto_instrument=True) # ← every LangGraph/LangChain call now traced

Notice what you did NOT do: add print statements, edit nodes, touch the agent at all. Instrumentation watches from outside — that's the standard (OpenTelemetry, the same tech that traces microservices).

🏢 Langfuse & LangSmith — the enterprise cousins

Same traces, same spans, same token counts — but hosted, with teams, dashboards, alerting and retention policies. Langfuse is open source (self-host or cloud); LangSmith is LangChain's own hosted product.

👨‍🏫 Watch now: your trainer will demo Langfuse Cloud live — the same trace tree you just built locally, in a hosted dashboard. The concepts transfer one-to-one; only the URL changes.

Interview line worth keeping: "traces and spans are OpenTelemetry concepts — Phoenix, Langfuse and LangSmith are different lenses on the same standard."

💰 Observability is where cost control starts

Tokens are money. Once every span shows its token count, cost stops being a monthly surprise and becomes a per-step number you can act on. In the lab you'll compute cost per triage, project it at Meridian's 50,000 triages/month — and then fix the embarrassing thing the trace exposes: policy_check pays to embed the same query on every single run. One small cache and every repeat lookup after the first is free. Found by reading a trace, not by guessing.

🤔 Predict: your agent gave a wrong answer on one run out of thousands. Without tracing, what's your realistic debugging move?
C — and that's the point. The code didn't change between the good runs and the bad one, and the customer won't remember exactly. Without a trace, a one-in-a-thousand failure is unreproducible folklore. With a trace, you open that run, read the exact prompt the model saw and the exact thing it returned, and the mystery usually dies in minutes.

Concept D
Evaluation: "It Looked Fine in the Demo" Is Not Evidence
A demo is one run on a friendly input. An eval is many runs on inputs chosen to hurt — scored, every time you change anything.
🎯
A golden set
A small, hand-written list of inputs where YOU already know the right behaviour. Unit tests for your agent — the lab's has 8 cases: normal, boundary, and two attacks.
⚖️
Deterministic checks first
"Did G2 route to human_approval?" is a plain ==. Always use == where == works — it's free and never hallucinates. That's routing accuracy.
🧑‍⚖️
LLM-as-judge for the rest
"Is this reply truthful to the policy?" has no ==. So a second model judges the first one's output against evidence, returning a structured PASS/FAIL + reason.

⚠️ Iron rule: never trust a judge you haven't seen catch a lie

A judge that passes everything is worse than no judge — it manufactures false confidence. So the lab does something sneaky first: it feeds the judge a planted lie ("the UPI limit is ₹5,00,000") against the real policy article, and only proceeds after the judge FAILS it. Calibrate the measuring stick before you measure.

📦 The two judged metrics — and where they come from

Faithfulness: does the reply contain only claims supported by the source? (The Air Canada test.) Answer relevance: does the reply actually address what was asked? These are the headline metrics of ragas, the open-source RAG/agent evaluation library. You'll hand-build both in ~20 lines — so you can see a judge is just a model + a rubric + structured output. When you want the industrial version:

pip install ragas from ragas.metrics import faithfulness, answer_relevancy, context_recall # same ideas, plus retrieval metrics, dataset tooling and CI integration

Finale · Industry Best Practice
The Readiness Review: Evidence, Not Vibes
No new concepts here — this is where the four layers pay off, in the exact format real engineering orgs use to decide "ship or don't."

📋 What a deployment-readiness checklist is

A list of claims about your system where every line carries its evidence — not "we added guardrails" but "the SYSTEM OVERRIDE attack was blocked before any LLM call, in Task 2, and here's the log line." Your notebook generates this automatically in Task 5, scoring 13 checks from things that actually happened in your run. If a check fails, the checklist says NO-GO and names it. That honesty is the feature.

1. Guardrail I proved: (one line — which one, and what it blocked) 2. Most expensive step: (from YOUR Phoenix trace — name + tokens) 3. Verdict: GO / NO-GO — and the one number that decides it

Scored on the three things a real review scores: guardrails, observability, and a credible deployment plan. Your checklist file is the handout behind your three lines.
