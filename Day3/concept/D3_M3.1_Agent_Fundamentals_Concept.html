What's An Agent, Really?
Three things that sound similar. They are not the same thing. Let's line them up.

🧭 Three ways to use an LLM — side by side

💬
A chatbot
One question in, one answer out. No memory of steps. No tools.
⛓️
A chain (Day 2)
A fixed sequence of steps, decided by you, the developer, in advance. The model fills in each step.
🤖
An agent (today)
The model decides which steps, in what order, and whether it needs a tool at all — using tools you give it, in a loop.

📐 The plain-English definition

A system is "agentic" when it has three things at once: autonomy (it decides its own next step, nobody hard-coded an if/else), tools (real actions it can take beyond just talking — search, calculate, call an API), and a feedback loop (it looks at what a tool returned and decides what to do next, instead of stopping after one shot).

Missing any one of those three, and it's not really an agent — it's a chatbot with extra steps, or a chain with a fancier name.

🤔 Quick guess: is M2.1's Concept D demo (the calculator-or-not decision) an agent by this definition?
Yes — barely. It had autonomy (the model chose) and one tool. What it didn't have was a real feedback loop, because one tool call was enough to answer and it stopped. Today you'll see the loop actually run more than once, when one tool's answer feeds the next decision.

Concept A
Think → Act → Observe → Think Again
This pattern has a name: ReAct (Reason + Act). Click through a real trace, one step at a time.

🔁 Mental model: a detective, not a script-reader

A script-reader says the same lines every time, in order. A detective looks at a clue, thinks about what it means, decides what to check next, checks it, then thinks again based on what they found — repeating until they have the answer. An agent works like the detective, not the script-reader.

One Description, One Decision
Give the agent a second tool. Watch it read both descriptions and pick — every single time, not once.

🏷️ The only thing the model sees: your tool's description

The model never sees your tool's code. It sees a name and a one-line description — "Evaluate a simple arithmetic expression" or "Search Meridian's policy knowledge base." Routing is really just the model matching a question against those descriptions, the same way you'd skim two folder labels to know which one to open.

Concept C
When One Tool Isn't Enough
Some questions need two tools, in order, where the second depends on what the first returned. This is planning.

🧩 The question: "A customer wants to send ₹40,000, ₹35,000 and ₹20,000 via UPI today. Will they breach the daily UPI limit, and by how much?"

Neither tool alone can answer this. The calculator doesn't know Meridian's UPI limit. The KB search doesn't add numbers. The agent has to plan: look the limit up first, then do the arithmetic against it.

The agent worked that ordering out for itself, from the question alone. That's the difference between this and a Day 2 chain — a chain's order is fixed at build time; an agent's order is decided at run time, per question.
🤔 If the KB search had come back with the WRONG limit, what would happen to the final answer?
A is right! This is worth sitting with: an agent's reasoning is only as good as what its tools hand back. A bad retrieval (wrong article, stale data) quietly poisons every step built on top of it — nothing here checks that the number itself is correct, only that the arithmetic is done correctly.

Concept D
The Most Important Slide Today: When NOT To Use One
Agents are slower, less predictable, and more expensive than a plain function call. Use one only when the job actually needs a decision.
Fixed, deterministic task → plain function / chainJudgment call, variable path → agent

✅ Good agent job

"Should I search the KB or calculate, or both, or neither?" — the right path genuinely depends on the question. That's a judgment call.

❌ Bad agent job

"Add these two numbers." There's exactly one correct way to do this. An agent here is a slow, unreliable, expensive way to call eval().

Try it — Same task, agent vs. plain function
Click each button. Watch the time and the round trip.
~1200 ms — one full LLM round trip to decide, then call the tool
🤔 For a fixed nightly batch job that always does the same 3 steps in the same order, should you build it as an agent?
B is right! "Advanced" isn't the goal — matching the tool to the job is. If the order never changes, a chain is faster, cheaper, and easier to test than an agent making the same decision over and over for no reason. Save agents for genuinely variable paths.
