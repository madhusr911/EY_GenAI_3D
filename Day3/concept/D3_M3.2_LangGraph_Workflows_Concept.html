Why Can't This Morning's Agent Do This?
M3.1's agent was genuinely smart. It was also, in three specific ways, fragile.
🧠💨
No memory between runs
Call it twice, it's a stranger both times. Nothing carries over.
⏸️🚫
No pause button
It runs start to finish in one breath. It can't stop mid-way and wait for you.
💥
No comeback after a crash
If the process dies mid-run, everything it had done is gone. Start over.

📐 What LangGraph actually adds

LangGraph doesn't replace your agent's reasoning — it gives it a shared notebook (state), a save file (a checkpoint) written after every step, and a pause button it can hit on purpose. That's the whole module: three upgrades, one at a time.

🤔 Quick guess: for a regulated task like approving a UPI transfer that breaches the daily limit, which of M3.1's three gaps matters most?
Arguably the pause button — a bank cannot let an AI system move large sums on its own, no exceptions. But you'll see today that all three connect: the pause button only works because there's a save file underneath it, and the save file is also what lets you recover from a genuine crash.

Concept A
A Flowchart You Actually Get to Run
Three pieces: state (the shared notebook), nodes (plain functions), edges (arrows between them).

📓 The shared notebook, passed hand to hand

Every node is a normal Python function. It reads the current state, does its job, and hands back only the fields it changed. LangGraph merges that into the shared state before the next node runs. No node ever sees another node's code — only what's written in the notebook so far.

builder = StateGraph(TriageState) builder.add_node("triage", triage_node) builder.add_node("policy_check", policy_check_node) builder.add_edge(START, "triage") builder.add_edge("triage", "policy_check") graph = builder.compile() # ← turns your boxes-and-arrows into something runnable

⚠️ One easy-to-miss rule: updates REPLACE, unless you say otherwise

If two nodes both return a value for amount, the second one simply overwrites the first — by default. That's fine for most fields. But for something you want to keep growing, like a running activity log, you tell LangGraph "add to this, don't replace it" using a small annotation called a reducer. You'll see this in the lab's log field — every node adds one line to it instead of wiping out the others.

Concept B
A Fork in the Road, Decided by the Graph Itself
One node can send execution down different paths depending on what's currently in state. That's a conditional edge.

🚦 Meridian's actual rule for this lab

UPI transfers that stay under the day's ₹1,00,000 limit auto-process. Anything that would push the customer OVER that daily limit needs a human's sign-off first. One function looks at state["amount"] and returns the name of whichever node should run next — nothing fancier than an if/else that the graph then follows.

def route_by_risk(state): if state["amount"] > 100000: return "human_approval" return "auto_process" builder.add_conditional_edges("policy_check", route_by_risk, {"human_approval": "human_approval", "auto_process": "auto_process"})

Concept C
A Save File, Written After Every Step
This is the single idea that makes everything else today possible.

🎮 Mental model: a video game save file

A checkpointer writes the shared state to disk after every node finishes — automatically, no extra code in your nodes. Close everything, reopen it later, and you can load that save file back and continue from exactly there. The save file's name is a thread_id — one per conversation, so Meridian can run thousands of these at once without them mixing up.

❌ In-memory checkpointer

Fast, zero setup — but the "save file" lives only in RAM. Restart the Python process and it's gone. Fine for a quick demo, not for a bank.

✅ SqliteSaver — what today's lab uses

A real file on disk. Kill the process, reopen a brand-new one pointed at the same file, and it remembers everything. "Durable" should mean something — this is what makes it true.

🤯
This is the flagship moment of today's lab
In Task 2, you will genuinely kill your Python process, start a fresh one, and watch it resume a paused workflow like nothing happened. Not simulated — a real restart.
🤔 Why does this matter more for a bank than for, say, a weather-chat demo?
Regulated workflows need an audit trail and crash recovery — a lending or payments decision can't just vanish because a server restarted. Durable execution is what makes "the workflow is still in progress, safely" a true statement instead of a hope.

Concept D
Pause for a Human, Then Resume Correctly
Today's required hands-on exercise. This is the pause button the Foundation section promised.

🛑 Mental model: a checkout that waits for a manager's approval

Big return at a store — the cashier can't just process it. The system freezes exactly there, a manager swipes a card, and only then does it continue. One line of code does this in LangGraph: interrupt(...) inside a node. Whatever you pass it becomes the question shown to the approver. The graph genuinely stops — not a fake pause, a real one, saved to the checkpoint.

def human_approval_node(state): decision = interrupt({"question": f"Approve this ₹{state['amount']:,.0f} UPI transfer — it breaches today's ₹1,00,000 limit?"}) return {"decision": decision} # ...later, once a human has decided... graph.invoke(Command(resume="approved"), config=config) # ← resumes, doesn't restart

👀 One more small thing: you can watch it happen live

While a workflow runs (and while it's paused, waiting), LangGraph can stream its state updates to you as they happen, instead of making you wait for one final answer. You'll see this in Task 3's output — the state prints as it changes, not all at once at the end.

Recover from a Crash — Not a Pause This Time

💥 The scenario: the notification service is down

Triage ran. Policy check ran. The human approved it. Then the very last step — texting the customer that it's done — throws an error because that outside service is briefly unreachable. In a system with no checkpoint, you'd have lost everything and would need to redo triage, policy check and the approval all over again. That would be a genuinely bad way to treat a customer's already-approved transfer.

Concept F · Stretch — Industry Best Practice
Rewind to Any Past Save Point, and Replay It Differently
Most teams building their first LangGraph workflow don't even know this exists. It's a genuine production debugging and audit tool.

⏪ Mental model: save points in a game, kept forever

Every single checkpoint your workflow ever wrote is still there, in order — not just the latest one. get_state_history() lists every one of them. Pick an earlier one — say, right before the human's decision — and you can resume from THAT point with a different answer. This is used in production to answer "what would have happened if we'd approved this instead of rejecting it?" without touching the real, original outcome.

⚠️ The one subtlety worth knowing

Just pointing at an old checkpoint and resuming isn't enough — LangGraph remembers that step already ran there and will hand you back the SAME old answer. To genuinely replay it differently, you first mark that checkpoint as a fresh branch point (one small call), then resume with your new decision. The lab notebook shows you exactly where this goes — it's the one place today where doing it the "obvious" way silently gives the wrong result, which is exactly why it's worth calling out.

A · Build the Graph

B · Branch on Risk
