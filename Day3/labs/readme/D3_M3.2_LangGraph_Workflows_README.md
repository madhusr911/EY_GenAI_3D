# D3 · M3.2 — LangGraph 1.0: Stateful, Durable Workflows

**Duration:** ~2 hrs · **Type:** Individual, run-and-understand · **Capstone:** this lab **is**
Phase 3 — a durable, human-supervised triage workflow, not a separate exercise standing in for it

## The situation

This morning's M3.1 agent could reason and pick tools, but it had no memory between runs, no
pause button, and no comeback after a crash. Today you wrap that same reasoning inside a
LangGraph workflow that fixes all three: a Meridian UPI-transfer triage workflow that pauses for
a human's approval before a transfer that breaches the daily limit, survives a real process
restart, recovers from a genuine failure, and can even be rewound to replay a past decision
differently.

This notebook is fully working code — nothing to write. Run each cell, and read what it prints.

## Work this side by side with the concept page, not start to finish

This module has two files, meant to be used together:

- `Day3/concept/D3_M3.2_LangGraph_Workflows_Concept.html` — seven short sections (Foundation,
  A, B, C, D, E, F)
- `Day3/labs/starter/D3_M3.2_LangGraph_Workflows_Starter.ipynb` — the notebook below

The pattern: read a concept section on the page → run the matching task(s) in the notebook →
come back to the page for the next concept → and so on. The notebook's own markdown cells tell
you exactly when to switch back to the page. Seven round trips, not one long read followed by
one long lab.

## Prerequisites

- Day 3 M3.1 completed — this lab reuses its exact `search_meridian_kb` approach over the same
  40 Meridian articles from Day 2's M2.3/M2.4 lab.
- `OPENAI_API_KEY` set as an environment variable (same `.env` file you've used all week).
- `student_config.json` created at the repo root with your real name and GitHub username.
- **New packages this module:** `langgraph` and `langgraph-checkpoint-sqlite` — installed by the
  Setup cell's `%pip install`, same as every other module's first cell.
- **Run this notebook from a normal local folder, not a cloud-synced one** (OneDrive / Dropbox /
  Google Drive). This lab creates a real SQLite file on disk to prove durability, and SQLite's
  file-locking does not work reliably inside a cloud-sync folder. If you see a "disk I/O error"
  from `SqliteSaver`, this is almost certainly why.

## How to work through this module

1. Open `Day3/concept/D3_M3.2_LangGraph_Workflows_Concept.html` in your browser and
   `Day3/labs/starter/D3_M3.2_LangGraph_Workflows_Starter.ipynb` in Jupyter or VS Code — side by
   side if your screen allows it.
2. Run the notebook's **Setup** cell first — it loads your API key, the Day 2 Meridian article
   data, and defines the shared model.
3. **Foundation + Concept A + Concept B → Task 1.** Read "Why Can't This Morning's Agent Do
   This?", "A Flowchart You Actually Get to Run" and "A Fork in the Road, Decided by the Graph
   Itself" on the page, then run Task 1's two cells: build the full triage graph and run a small
   (auto-process) and a large (needs-approval) request through it — with no checkpointer yet, so
   you'll also see exactly why trying to resume the paused one fails.
4. **Concept C → Task 2.** Read "A Save File, Written After Every Step", then run the cell that
   attaches a real `SqliteSaver`, pauses a large request, and proves persistence for real by
   building a brand-new graph object pointed at the same file.
5. **Concept D → Task 3 (required exercise).** Read "Pause for a Human, Then Resume Correctly",
   then run the cell that streams intermediate state live and resumes two separate paused
   requests — one approved, one rejected.
6. **Concept E → Task 4.** Read "Recover from a Crash — Not a Pause This Time", then run the
   cell that forces a real node failure and recovers from it on the same thread_id.
7. **Concept F → Tasks 5-6 (stretch / Diamond).** Read "Rewind to Any Past Save Point, and
   Replay It Differently", then run the cells that fork Task 3's approved conversation from its
   checkpoint history and replay it as a rejection, and finally save your results.
8. Read the Recap section on the page, then confirm the notebook's final cell has run — it saves
   `m3_2_langgraph_results.json` next to the notebook.
